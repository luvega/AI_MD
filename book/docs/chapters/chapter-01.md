# 第 1 章 Linux 与生化计算基础

## 本章导读

本章是整套课程的运行底座。后面的 PyMOL/Chimera、分子对接、MD、Boltz2、RFdiffusion/RFD3、Chai-1 和 AI Agent 工作流，看起来分别属于结构生物学、计算化学、蛋白设计和软件工程，但它们在项目里共享同一组基本能力：知道文件在哪里，知道命令从哪里运行，知道输入是什么，知道输出是否完整，知道失败日志应该怎样保存。

对于药物化学研究者来说，Linux 不应该被理解成一个抽象的操作系统课程，而应该被理解成“计算实验台”。在这个实验台上，FASTA、PDB、mmCIF、SDF、SMILES、YAML、CSV、TSV 和日志文件对应真实实验中的样品、标签、仪器参数和原始记录。命令行不是为了炫技，而是为了让一次结构准备、一次 docking、一次 MD 或一次 protein design 可以被复查、复跑和复用。一个没有记录命令、版本、输入和输出路径的计算结果，即使图片很好看，也很难进入严肃研究链条。

本项目的第一版在线书籍已经建立了章节导航；第二版从本章开始把“导航”变成“课程讲义”。本章的重点不是要求读者背熟所有 Linux 命令，而是建立三条习惯：第一，所有计算都必须从明确的项目目录开始；第二，所有中间文件都必须能追溯到原始输入和参数；第三，所有可长期复用的信息都应进入 wiki、方法卡或实验记录，而不是只留在聊天窗口、临时下载目录或桌面截图里。

## 学习目标

完成本章后，读者应能把 Linux 基础操作转化为后续 AIDD 任务的可复现准备工作。具体来说，读者需要能解释工作目录、绝对路径、相对路径、环境变量、conda 环境、文件权限和日志重定向在计算药物化学中的作用；能区分原始资料、知识库笔记、方法卡、实验记录、运行输出和临时缓存；能为一个最小计算任务建立 `inputs/`、`outputs/`、`logs/`、`scripts/` 和 `notes/` 目录；能检查 FASTA、PDB/mmCIF、SDF、SMILES、YAML、CSV/TSV 等文件是否符合下游软件要求；能把一次失败运行记录成可诊断的问题，而不是简单写成“软件报错”。

更高一级的目标是形成研究工作台意识。很多计算任务失败并不是因为模型本身不可用，而是因为路径写错、格式不一致、链 ID 混乱、配体名称不一致、GPU/CPU 环境不匹配、依赖混装、输出被覆盖或日志没有保存。本章训练的是这些“低层错误”的防御能力。只有这些底层习惯稳定，后续章节才有资格讨论 docking score、MD 稳定性、Boltz2 亲和力、RFD3 binder 设计和 Chai-1 aggregate score 的科学含义。

## 核心概念

工作目录是所有命令的坐标原点。很多初学者在 Windows 文件管理器中可以找到文件，但在 Linux shell 中找不到，是因为没有意识到命令解释器总是从当前目录解析相对路径。`pwd`、`ls`、`cd`、`mkdir`、`cp`、`mv`、`rm`、`find`、`grep` 或 `rg` 看似普通，却决定了后续几乎所有工具能否找到输入。对于 AI_MD 这类项目，应始终先确认当前目录是项目根目录，再运行验证脚本、构建书籍或读取知识库文件。

绝对路径和相对路径需要分清。绝对路径从盘符或根目录开始，适合记录长期位置；相对路径从当前目录开始，适合项目内部链接和脚本复用。在线书籍和 wiki 中优先使用项目相对路径，是为了让仓库换盘、复制或部署到 GitHub Pages 后仍然可导航。但实际运行大型计算时，日志里也应记录真实运行目录和绝对输出路径，防止不同机器之间混淆。

环境变量是软件查找依赖、模型、缓存和可执行程序的隐性配置。`PATH` 决定命令能否被找到，`CUDA_HOME`、`LD_LIBRARY_PATH` 或工具特定变量决定 GPU 和动态库能否被调用。对计算药物化学项目来说，环境变量错误常常表现为“同一命令在一台机器能跑，在另一台机器不能跑”。因此每次建立新环境，都应该记录操作系统、Python 版本、conda 环境名、关键包版本、GPU/CPU 信息和安装来源。

文件格式是生化计算的共同语言。FASTA 和 A3M 主要承载序列与 MSA；PDB 和 mmCIF 承载三维结构、链、残基、配体、金属离子和晶体学信息；SDF、MOL2、SMILES 和 InChI 承载小分子结构；PDBQT 常见于 AutoDock/Vina 系列；DCD、XTC、TRR 和 NetCDF 承载 MD 轨迹；YAML/JSON 常用于 AI 模型输入和结果汇总；CSV/TSV 负责批量筛选表格。研究者要能看懂这些文件的角色，而不是只凭扩展名传给下游软件。

目录分层是知识库和实验记录的安全边界。AI_MD 项目已经把 `06_原始学习素材/` 作为只读原始资料层，把 `01_课程章节索引/`、`02_方法笔记/`、`03_文献笔记/` 和 `07_研究工作台/` 作为整理后的 wiki 层，把 `04_实验记录/` 作为运行记录层，把 `book/` 作为在线书籍出口层。这样的分层能避免把原始 PDF、课程范文、方法解释和本项目运行结果混写在一起。后续章节反复强调的“文献案例不是本项目结果”，正是从这套目录纪律开始的。

可复现记录不是额外负担，而是计算实验的最低质量线。一次可复现记录至少包括输入文件、输入来源、软件名称、软件版本、命令或配置、随机种子、运行时间、输出目录、关键结果、失败信息和人工判断。对于 AI 模型，还应记录模型名称、checkpoint、权重来源、API/网页版本、参数、batch 大小和是否使用模板/MSA/约束。没有这些字段，后续很难判断一个分数变化来自真实科学差异，还是来自环境、版本和输入处理差异。

## 方法流程

第一步是建立项目目录。一个最小计算任务可以采用 `inputs/`、`outputs/`、`logs/`、`scripts/`、`env/` 和 `notes/` 的分层。`inputs/` 保存本次运行实际使用的输入副本或输入清单，`outputs/` 保存工具生成结果，`logs/` 保存标准输出、错误输出和运行摘要，`scripts/` 保存命令脚本或解析脚本，`env/` 保存环境导出文件，`notes/` 保存人工判断。这样的结构可以直接映射到 `04_实验记录/` 模板。

第二步是检查输入。对蛋白序列，要确认 FASTA header、序列长度、非标准氨基酸和换行；对结构文件，要确认链 ID、缺失残基、插入码、配体名、金属离子、水分子、altloc 和坐标单位；对小分子，要确认质子化状态、电荷、互变异构、手性、盐形式和 3D 构象；对表格，要确认分隔符、列名、重复 ID、空值和编码。不要把“文件能打开”当成“文件可用于计算”。

第三步是建立独立软件环境。不同工具对 Python、CUDA、PyTorch、OpenMM、RDKit、OpenBabel、NumPy 或编译器版本要求不同，混装会造成难以诊断的错误。保守策略是为 docking、MD、结构预测、蛋白设计和数据分析分别建立环境，并导出 `environment.yml` 或 `requirements.txt`。如果工具通过网页或云平台运行，也要记录网页版本、上传文件、参数截图或下载包 hash。

第四步是小样例验证。任何大型任务都应该先跑最小输入。例如在 docking 前先用一个受体和一个配体确认格式转换、box 和输出；在 MD 前先用短时间平衡确认参数化和能量稳定；在 Boltz2 前先用一个简单 YAML 确认链和配体可被解析；在 RFdiffusion 或 RFD3 前先用少量候选确认 contig、hotspot 和输出结构合理。小样例成功并不证明科学结论成立，但能快速发现路径、依赖和格式问题。

第五步是运行与日志。建议把命令写入脚本，运行时同时保存标准输出和错误输出。对长任务，日志里应能看到开始时间、结束时间、命令、环境、输入、输出和退出状态。对批量任务，应生成 manifest 表，记录每个样本的状态、错误、输出文件和是否进入下一轮分析。这样后续章节的虚拟筛选、MD 批处理、Chai-1 PPI 面板和蛋白设计候选筛选才不会变成不可追踪的文件堆。

第六步是结果归档。临时输出不应直接写入课程正文；应先进入实验记录模板，确认哪些是原始输出，哪些是解析结果，哪些是人工判断，哪些只是失败记录。长期有价值的流程再沉淀为方法卡，有引用价值的论文再进入 Zotero/BibTeX，有教学价值的说明再进入在线书籍。这个流向能保证知识库既能服务 AI 检索，也能服务个人研究。

## 实验/练习入口

练习一：建立一个空白计算任务目录。目录名采用日期加任务名，例如 `2026-05-31_docking-dry-run/`。在其中创建 `inputs/`、`outputs/`、`logs/`、`scripts/` 和 `notes/`。把一个 FASTA、一个 PDB 或一个 SDF 文件放入 `inputs/`，在 `notes/README.md` 中写明来源、用途和是否可公开。这个练习不要求跑任何复杂软件，目标是让读者习惯先建实验容器，再开始计算。

练习二：写一个输入检查清单。对 FASTA，检查序列长度、非标准字符和 header；对 PDB/mmCIF，检查链 ID、配体、缺失残基和水分子；对 SDF/SMILES，检查 ID、质子化、电荷和 3D 构象；对 CSV/TSV，检查列名、分隔符、重复 ID 和空值。把检查结果写成表格，并标出“可以进入下游”“需要人工修正”“暂不使用”三类状态。

练习三：模拟一次可复现运行记录。可以不真正运行大型软件，只写一个 dry-run 记录：任务目标、输入、命令、参数、预期输出、可能失败点、日志路径和下一步判断。然后把这份记录与 [附录 B](../appendices/experiment-templates.md) 中的模板对照，补齐缺失字段。这个练习能直接衔接第 3 章 docking dry-run 和第 5 章 Boltz2 结果归档。

## 使用边界与常见误读

第一，Linux 命令运行成功不等于科学结果可信。命令成功只说明程序完成了某个计算步骤，不能替代输入质量、参数合理性、模型适用性和结构复核。后续章节里 docking score、RMSD、Boltz2 affinity、RFdiffusion 输出和 Chai-1 aggregate score 都必须放在具体模型边界内解释。

第二，复制路径不等于建立 provenance。把某个 PDF、PDB 或 CSV 的路径写在笔记里，只能说明文件位置；要形成 provenance，还需要来源、日期、版本、处理步骤、是否人工修改、是否为原始文件、是否进入 Zotero/BibTeX 或实验记录。AI_MD 项目尤其强调原始资料只读，在线书籍不直接复制原始 PDF 内容。

第三，环境装好不等于可复现。真正可复现还需要环境导出、版本锁定、模型权重来源、随机种子、输入 manifest 和输出校验。很多 AI 工具网页版本变化很快，如果只记录“用了某某网站”，半年后几乎无法复盘。

第四，不要把所有东西都写进正文。在线书籍负责讲授概念、流程和边界；wiki 负责沉淀知识；实验记录负责保留具体运行；原始素材负责保存来源；Zotero/BibTeX 负责引用权威。把这些层混在一起，会让 AI Agent 和人工读者都难以判断一段话是课程解释、文献证据还是本项目结果。
